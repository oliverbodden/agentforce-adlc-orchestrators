# Target Structure Review

## Purpose

This document proposes the Phase 0 target structure before implementation changes. It must be approved before Phase 1 overlays, docs, installer, or skill changes begin.

## Proposed Structure

```text
Salesforce upstream repo
  -> standard consolidated skills
  -> source of truth for standard ADLC skill behavior

Project/local overlay source
  -> wrappers, additive guidance, project playbooks, bootstrap, and version manifests
  -> must not modify or delete Salesforce standard skill content directly

Developer workstation
  -> installed Salesforce upstream skills
  -> installed Project/local overlay
  -> setup/status report proving versions and remotes

Corporate artifact repo
  -> ticket artifacts, HITL logs, eval reports, baselines, rollups
  -> canonical repo: https://github.com/YOUR_ORG/YOUR_ADLC_ARTIFACT_REPO

Production agent repos/orgs
  -> agent metadata, deployable code, Salesforce org state
  -> not the place for raw ADLC run artifacts unless explicitly approved
```

## Layer Map

| Layer | Proposed owner/source | Contents | Guardrails |
|---|---|---|---|
| Salesforce upstream | `SalesforceAIResearch/agentforce-adlc` | `developing-agentforce`, `testing-agentforce`, `observing-agentforce`, assets, hooks, shared scripts, installer | Treat as standard. Do not edit/delete directly. |
| Project/local overlay | Corporate ADLC source repo to be confirmed, likely `YOUR_ORG/YOUR_ADLC_ARTIFACT_REPO` or a sibling overlay repo | `adlc-drive`, `adlc-execute`, `adlc-ticket`, add-only patches, project playbooks, version manifests, bootstrap | Additive only. Use markers, owners, source refs, and conflict checks. |
| Artifact store | `https://github.com/YOUR_ORG/YOUR_ADLC_ARTIFACT_REPO` | Goals, originals, specs, attempts, HITL, eval reports, baselines, rollups, monthly reviews | Developers submit per-ticket artifacts. Rollups generated by CI/maintainers. |
| Developer machine | Local clone plus `~/.cursor/skills` or Claude plugin install | Installed standard skills plus local overlay and setup report | Bootstrap must be dry-run/status capable and non-destructive by default. |
| Production agent code/org | Existing agent source repos and Salesforce orgs | Agent metadata, `.agent` files if used, Flow/Apex/config, deployed versions | Do not push raw ADLC artifacts here by default. |

## Artifact Flow

```text
JIRA / ticket / goal
  -> local ticket intake (`adlc-ticket` / wrapper)
  -> local drive/execute process
  -> upstream consolidated skills for build/test/observe tasks
  -> ticket artifacts in corporate artifact repo
  -> HITL rollups and monthly skill-improvement review
```

## Installation Flow

Proposed clean workstation flow:

```text
1. Clone corporate ADLC repo
2. Run bootstrap dry-run
3. Install/verify prerequisites, including Salesforce CLI and required command/plugin areas
4. Install Salesforce upstream ADLC baseline
5. Apply Project/local overlay additively
6. Verify installed skills, remotes, versions, and artifact paths
7. Produce local setup report
```

The bootstrap must refuse destructive overwrites unless explicitly invoked with a reset flag.

## Salesforce CLI Requirements

The Salesforce upstream installer treats Salesforce CLI as optional because the skills can be installed without an org connection. For our developer workflow, Salesforce CLI and the Agentforce command surface are required for the skills to work correctly.

Bootstrap/status must verify:

| Requirement | Why it matters |
|---|---|
| `sf` is installed and on PATH | All build/test/observe skills call Salesforce CLI commands. |
| `sf --version` meets approved minimum | Upstream testing docs mention `sf agent preview` trace issues on older versions and recommend current CLI. |
| `sf agent` is available | Required for generate, validate, preview, publish, activate, and Testing Center commands. |
| `sf project` is available | Required for deploy/retrieve of metadata and authoring bundles. |
| `sf data` is available | Required for SOQL/org checks and metadata-related queries. |
| `sf apex` is available | Required for STDM/helper Apex execution and diagnostics. |
| `sf api request` is available | Required for Data Cloud data-space discovery and REST calls. |
| Required org aliases are authenticated | Preview, Testing Center, Tooling/API calls, and deploy flows need org auth. |

Current local reference state captured during Phase 0:

```text
@salesforce/cli 2.130.9
agent 1.32.16 (core)
api 1.3.14 (core)
apex 3.9.17 (core)
data 4.0.85 (core)
deploy-retrieve 3.24.23 (core)
org 5.9.79 (core)
```

The bootstrap should not only check that `sf` exists. It should run command availability checks such as:

```text
sf agent --help
sf project --help
sf data --help
sf apex --help
sf api --help
```

If a required command surface is missing, onboarding must stop with remediation instructions before declaring the machine ready.

## Required Approval Decisions

Before Phase 1 implementation, approve or revise:

1. Whether `SalesforceAIResearch/agentforce-adlc` is the authoritative upstream baseline.
2. Whether `YOUR_ORG/YOUR_ADLC_ARTIFACT_REPO` is both the local overlay source and artifact store, or artifact store only.
3. Whether `adlc-drive` and `adlc-execute` remain local wrapper skills or should be reworked into docs/playbooks around the three consolidated Salesforce skills.
4. Whether legacy patches become patch files, wrapper instructions, project playbook guidance, or a mix.
5. Whether local Tooling API instruction edits remain allowed for UI-built agents, given possible tension with upstream `.agent` source-of-truth guidance.
6. What minimum Salesforce CLI version and plugin/command surfaces should be required for project developers.

## Recommendation

Proceed with an additive overlay model:

- Salesforce upstream is standard baseline.
- Local `adlc-drive`, `adlc-execute`, and `adlc-ticket` remain wrapper/process overlays for now.
- Local project ADLC docs/reporting/HITL/artifacts remain project-owned.
- Legacy base-skill patches are not blindly re-applied; each is remapped and reviewed against the consolidated upstream skills.
- No Phase 1 implementation starts until this target structure is approved.
