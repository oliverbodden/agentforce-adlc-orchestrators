# ADLC Artifact Repo Workflow

Local guidance for storing Agentforce ADLC run artifacts in the shared shared artifact repo.

Canonical artifact repo:

```text
https://github.com/YOUR_ORG/YOUR_ADLC_ARTIFACT_REPO
```

This repo is separate from production agent/code repos unless written governance explicitly says otherwise.

---

## Repository Routing Guardrail

Before closeout, verify the target repo.

Expected prompt:

```text
I need to submit ADLC artifacts to the shared artifact repo, not the production agent repo.

Expected artifact repo: https://github.com/YOUR_ORG/YOUR_ADLC_ARTIFACT_REPO
Current git remote: <detected-remote-url>
Target branch: <branch>
Ticket folder: adlc/agents/<agent>__<org>/tickets/<ticket>/

Proceed with artifact submission, or open/clone the correct artifact repo first?
```

Rules:

- Do not assume the current workspace is the artifact repo.
- If the current git remote is a production agent repo or a developer's feature repo, stop.
- Guide the user to clone/open the shared artifact repo separately.
- Never push ADLC artifacts into a production agent repo unless governance explicitly says that repo is also the artifact repo.
- Each run writes to a unique ticket folder.
- Existing ticket folders are append-only unless the user confirms they are resuming that exact run.
- Individual developers submit per-ticket artifacts. Shared rollups should be generated by CI or a controlled maintainer process where possible.

---

## Artifact Layout

Recommended layout:

```text
adlc/
  agents/
    <agent>__<org>/
      tickets/
        <ticket>/
          goal.md
          hitl.jsonl
          discovery.json
          config.json
          status.md
          dependency-map.md
          originals/
          specs/
            regression.yaml
            feature.yaml
            scenarios.yaml
          attempts/
            01-<short-name>/
          eval-report.json
          eval-report.html
      baselines/
  rollups/
    hitl-index.jsonl
    ticket-index.jsonl
    monthly/
      YYYY-MM-skill-improvement.md
```

Local project folders may keep the same per-ticket structure during execution. Closeout should copy or submit the final artifact package to the shared repo.

---

## Minimum Closeout Package

Before a run is considered closed, verify:

- `goal.md` exists and states scope, assumptions, and final status.
- `hitl.jsonl` exists and is valid JSONL.
- `discovery.json` exists after Phase 3a, or the reason it is missing is logged. It records durable machine-readable handoff state such as source of truth, canonical ticket folder, instruction IDs, baseline CSV, config path, Testing Center suite name, and diagnostic mode if used.
- `config.json` exists or the reason it is missing is logged.
- `dependency-map.md` or equivalent dependency map exists.
- `originals/` exists for any changed instruction/configuration.
- `specs/` exists for regression, feature, or scenario tests used.
- `attempts/` contains the final candidate and relevant intermediate attempts.
- `eval-report.json` and `eval-report.html` exist, or a logged reason explains why eval was not run.
- `status.md` records `GO`, `NO-GO`, `CONDITIONAL`, or `ABANDONED`.
- Any `skill-flaw`, `local-overlay-flaw`, `upstream-standard-gap`, `upstream-conflict`, or `installation-drift` has `improvement_signal: true`.
- Environment/testability caveats and residual risk owners are recorded.

Do not use `.adlc-drive-state.json` as a closeout artifact. It is a local resumability scratchpad only; durable audit and handoff state must be represented in `hitl.jsonl`, `discovery.json`, `config.json`, and final reports.

## HITL Rollups

Monthly AI review should use rollups generated from per-ticket artifacts:

```text
rollups/hitl-index.jsonl
rollups/ticket-index.jsonl
rollups/monthly/YYYY-MM-skill-improvement.md
```

Rollup generation should classify recurring issues:

- Repeated user corrections.
- Wrong strategy selection.
- Missed hard gates.
- Weak eval criteria.
- Avoidable testing noise.
- Poor artifact quality.
- Salesforce upstream conflicts.
- Installation drift.

Monthly output should be a reviewed skill-improvement backlog, not automatic edits to production skills.
